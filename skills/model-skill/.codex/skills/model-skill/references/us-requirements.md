# 需求分析

## 适用场景

- 收到新功能需求或 PRD 文档
- 需要评估改动影响范围
- 需要设计新功能或扩展现有模块

## 分析步骤

### 1. 理解需求

确认以下信息：
- 影响哪些功能模块或子系统？
- 改动属于哪个层级（界面层 / 业务层 / 通信层 / 驱动层等，以本模块架构为准）？
- 是新增还是修改现有逻辑？

### 2. 影响范围评估

> 本模块的分层架构见 `CLAUDE.md` 的"架构概览"章节，以其为准评估改动波及范围。

核心问题：
- 该改动是否影响对外接口？
- 是否影响关联模块（见 `.evospec/module.config.yaml` → `related_modules`）？
- 是否需要同步修改板端配置文件（见 `module.config.yaml` → `board.config_file`）？

### 3. 接口 / 模块设计要点

- 新增对外接口 → 明确函数签名，更新对应头文件
- 新增消息 / 事件 → 定义消息 ID，确认收发双方
- 新增配置项 → 确认配置文件路径（`board.config_file`），明确字段格式

### 4. 输出设计方案（必须生成文档）

在 `.evospec/output/design/` 下生成设计文档。

**文件命名**：`YYYYMMDD-<ones-id>-<简短描述>.md`
例：`20260509-701234-新增温控功能模块.md`

**文档模板**：

```markdown
# 设计方案

- 日期：YYYY-MM-DD
- ONES ID：#XXXXXX
- 需求描述：（一句话概括）

## 影响范围

| 层级 | 改动文件 | 改动类型 |
|------|----------|----------|
| （按本模块实际层级填写） | `路径/文件名` | 新增 / 修改 |

## 接口变更

（新增 / 修改的类、函数签名，无则填"无"）

## 配置变更

（配置文件新增字段，无则填"无"）

## 预估工作量

（如：2天）
```

设计文档生成后，进入功能开发：`@references/us-feature-dev.md`
