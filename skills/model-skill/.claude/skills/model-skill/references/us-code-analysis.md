# 代码架构分析

代码分析结果持久存放在 `.evospec/output/code-analysis/`，按模块拆分。
**bug排查时优先读对应分析文档，而不是重新扫描源码。**

## 分析文档索引

| 文档 | 覆盖内容 | 适用场景 |
|------|----------|----------|
| `core-modules.md` | 各核心类的职责、关键成员、调用关系 | 不确定改哪个文件时 |
| `upgrade-flow.md` | 六步升级流程每步的代码路径 | 升级卡在某阶段时 |
| `plugin-system.md` | 插件加载机制、dlopen流程、回调注入 | 插件未加载/回调未触发时 |
| `message-flow.md` | 总线消息 → daemon → core → plugin 完整链路 | 消息未收到/响应异常时 |
| `error-codes.md` | 所有错误码定义和含义 | 日志出现错误码时 |

## 使用方式

### bug排查时

1. 根据故障现象判断涉及哪个模块/流程
2. 读对应分析文档（如升级卡住 → 读 `upgrade-flow.md`）
3. 定位到可疑代码位置后，再按需读源文件确认细节

### 分析文档更新时机

以下情况需更新对应分析文档，保持知识库准确：
- 修改了核心流程逻辑（更新 `upgrade-flow.md`）
- 新增/修改插件（更新 `plugin-system.md`）
- 新增消息 ID 或改变消息路由（更新 `message-flow.md`）
- 新增错误码（更新 `error-codes.md`）

## 生成/更新分析文档

当用户说"分析代码"、"更新代码分析"或某文档已过期时：

1. 读取对应源文件（见各文档头部的"源文件"列表）
2. 按文档模板格式输出分析结果
3. 覆盖写入 `.evospec/output/code-analysis/<文档名>`
4. 在文档头部更新"最后分析时间"

## 新增分析文档

当现有文档不覆盖某个需要反复查阅的模块时，创建新文档：

**文件命名**：`<模块名>.md`

**文档头部格式**：
```markdown
# <模块名> 分析

- 最后分析时间：YYYY-MM-DD
- 源文件：`src/xxx/xxx.cpp`, `include/xxx.h`
```
