# 诊断：从对话里归纳根因

目标：把每一处"用户纠正"映射到 **trigger / flow / detail / template** 中的一类。分类决定下一步改哪个文件，错分类 = 改错地方。

---

## 四类根因速查

| 分类 | 含义 | 信号 |
|---|---|---|
| **trigger** | skill 本身该不该被触发出了问题 | "这种场景不用调这个 skill" / "你怎么没用 xxx skill" / skill 触发了但用户第一反应是"换一个做法" |
| **flow** | skill 触发对了，但步骤/顺序/判断条件有问题 | "先做 B 再做 A" / "漏了一步 X" / "不要每次都问我 Y" / "这种情况直接跳过 Z" |
| **detail** | 步骤对，但某一步里的细节偏了（技术栈、术语、约定） | "我们这边 ViewModel 叫 VM 不叫 VM­odel" / "Linux 下的路径要用正斜杠" / "这个协议字段的单位是 ms 不是 s" |
| **template** | 产出的默认模板、配置、示例不对 | "这个 commit 模板我们不用 emoji" / "config.yaml 默认 stack 应该是 linux/cpp" / "生成的文档结构不对" |

---

## 启发式问题（按顺序问自己）

1. **用户第一次纠正的是什么？**
   最早的纠正最能反映 skill 的结构性问题；后续纠正常常是被第一次纠正牵连出来的。

2. **用户有没有说"以后都这样" / "记住" / "下次不用问了"？**
   有 —— 这是**长期规则**信号，务必沉淀。
   没有 —— 可能只是本次会话的特殊要求，**不要**动 skill，提醒用户"这条看起来是一次性的，要我沉淀吗？"

3. **纠正是关于"做什么"还是"怎么做"？**
   - "做什么" 错了 → 偏 trigger 或 flow
   - "怎么做"（命名、格式、单位、模板）错了 → 偏 detail 或 template

4. **如果把这条纠正写成一句规则，它对**其他**用户也成立吗？**
   - 成立 → 沉淀到全局 skill
   - 只对当前用户/项目成立 → 建议用户放到项目级 skill 或 CLAUDE.md，不要污染全局 skill

5. **纠正背后有没有"为什么"？**
   有 —— 把 WHY 一起写进改动（LLM 能举一反三）。
   没有 —— 主动问用户一句："为什么这样更好？"。没有 WHY 的规则迟早被其他规则覆盖。

---

## 反面信号：不应该改 skill

遇到以下情况，**不要**动 skill，告诉用户走别的路径：

- 用户只是这次不想要 X，没说以后 → 一次性输出调整即可
- 用户抱怨的是环境 / 数据问题，而不是 skill 行为 → 是 bug 不是规则
- 用户明确说"这个跟 skill 无关"
- 纠正内容是项目级的（只对当前代码库成立） → 引导去改项目级 `CLAUDE.md` 或项目 `.claude/skills/`
- 纠正内容是个人偏好（措辞、语气） → 引导去写 memory 条目

---

## 输出格式（写进主 SKILL 的 Step 2 归纳里）

```
纠正 #1
  原输出  : {skill 最初给出的内容 / 做法}
  用户改成: {最终满意的版本}
  根因    : {一句话说明为什么 skill 会这样做}
  分类    : trigger | flow | detail | template
  WHY     : {用户给的理由，或你推断后让用户确认的理由}
  普适性  : 全局 / 项目级 / 一次性
```

普适性 = "一次性" 的条目直接跳过，不进入 Step 3。
