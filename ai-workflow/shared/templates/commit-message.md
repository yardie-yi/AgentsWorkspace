# Commit Message 模板（中文优先）

## 目标

生成清晰、具体、可读性强的 Git 提交信息。

该模板主要服务于如下工作流：

- AI 在工作分支上完成 bug 修复或需求开发
- 每完成一个独立主题后，生成一个可读的 commit
- 后续用户会从候选 commit 列表中挑选要合入主分支的提交

因此，提交信息必须满足：

1. 中文优先
2. 一眼看懂本次改动做了什么
3. 适合后续在候选列表中阅读和挑选
4. 一个 commit 只表达一个主题

---

## 标准格式

推荐格式：

`type(scope): 中文摘要`

示例：

- `fix(can): 修正收发器待机引脚控制逻辑`
- `fix(bsp): 修复启动阶段空指针访问`
- `feat(app): 增加设备状态上报接口`
- `refactor(uart): 拆分初始化与恢复流程`
- `chore(build): 调整调试版本编译参数`

---

## type 说明

- `fix`：缺陷修复
- `feat`：新增功能
- `refactor`：重构，不改变外部行为
- `chore`：构建、脚本、配置、杂项维护
- `docs`：文档
- `test`：测试
- `perf`：性能优化

---

## scope 说明

scope 应尽量体现改动所属模块，使用简短英文标识。

常见示例：

- `can`
- `uart`
- `bsp`
- `boot`
- `app`
- `middleware`
- `driver`
- `build`
- `net`

如果无法明确模块，可以省略 scope，例如：

- `fix: 修正调试版本下的启动异常`

但优先建议带上 scope。

---

## 中文摘要写法要求

摘要应直接说明“改了什么”，必要时体现“为什么改”。

推荐使用以下动词开头：

- 修正……
- 修复……
- 增加……
- 调整……
- 拆分……
- 限制……
- 优化……
- 避免……
- 统一……
- 补充……

### 推荐示例

- `fix(can): 修正收发器待机引脚控制逻辑`
- `fix(uart): 修复初始化超时判断错误`
- `fix(bsp): 避免启动阶段空指针访问`
- `feat(app): 增加设备状态上报接口`
- `refactor(boot): 拆分启动初始化流程`
- `chore(build): 调整调试版本编译开关`

---

## 必须遵守的原则

### 1. 一个 commit 只表达一个主题

该规则非常重要。

因为后续用户会从候选 commit 列表中挑选要合入主分支的提交，所以每个 commit 必须具备明确边界：

- 一个 bug 对应一个 commit
- 一个独立需求对应一个 commit
- 一个明确的重构动作对应一个 commit

不要把多个独立主题揉进同一个 commit。

### 2. 标题必须具体，便于后续挑选

提交标题必须让人一眼看懂本次改动的核心内容。

好的标题应该让用户在候选列表中，仅靠标题就能大致判断：

- 这个 commit 解决了什么问题
- 它属于哪个模块
- 是否值得现在合入主分支

### 3. 如果改动混合多个主题，优先建议拆分 commit

如果一次改动同时包含：

- bug 修复
- 日志调整
- 配置修改
- 小范围重构
- 注释清理

不要勉强生成一个大而全的标题，而应优先建议拆分为多个 commit。

---

## 禁止使用的空泛标题

不要生成以下风格的提交信息：

- 修改代码
- 更新代码
- 更新一下
- 修复问题
- 调整逻辑
- 临时处理
- 优化代码
- bugfix
- try fix
- fix again
- 修改若干问题
- 修复若干问题并优化日志

这些标题存在以下问题：

- 信息量太低
- 无法体现模块
- 无法体现改动目的
- 不利于后续在候选列表中选择

## 推荐写法模板

### 修 bug

- `fix(can): 修正收发器待机引脚控制逻辑`
- `fix(bsp): 修复启动阶段空指针访问`
- `fix(uart): 修正初始化超时判断条件`
- `fix(driver): 避免异常路径下重复释放资源`

### 新功能

- `feat(app): 增加设备状态上报接口`
- `feat(net): 支持链路断开后的自动重连`
- `feat(middleware): 增加消息重试机制`

### 重构

- `refactor(boot): 拆分启动初始化流程`
- `refactor(app): 统一设备状态处理入口`
- `refactor(uart): 分离初始化与恢复逻辑`

### 构建/脚本/配置

- `chore(build): 调整调试版本编译参数`
- `chore(ci): 更新构建脚本中的产物路径`
- `chore(config): 清理无效调试开关`

### 文档

- `docs(can): 补充驱动初始化说明`
- `docs(debug): 更新调试版本发布步骤`

### 测试

- `test(can): 增加收发异常场景测试`
- `test(uart): 补充初始化失败用例`

---

## 长提交信息模板（可选）

当改动较复杂，且确实需要补充说明时，可使用：

标题：
`type(scope): 中文摘要`

正文可选包含：

- 背景：为什么要改
- 改动：主要改了哪几处
- 影响：可能影响哪些模块
- 验证：做了什么验证

示例：

`fix(can): 修正收发器待机引脚控制逻辑`

- 背景：CAN 总线无法正常收发，怀疑收发器未正确退出待机
- 改动：修正 STB 引脚控制时序，补充初始化状态判定，清理错误注释
- 影响：CAN 驱动初始化与收发流程
- 验证：完成环回测试与基础收发测试

---

## 自动生成规则

生成提交信息时，应遵循以下优先级：

1. 优先准确表达改动本质
2. 优先体现模块 scope
3. 优先使用中文摘要
4. 优先让标题具备候选列表可读性
5. 如果改动主题不单一，优先建议拆分 commit，而不是硬写一个笼统标题

---

## 正式输出前建议

在真正执行 commit 前，建议至少给出：

- 建议标题
- 如果合适，可给出 2～3 个备选标题
- 如果当前改动主题不单一，应明确提示“建议拆分为多个 commit”

示例：

建议提交标题：
- `fix(can): 修正收发器待机引脚控制逻辑`

备选：
- `fix(can): 修复 STB 控制异常导致的收发失败`
- `fix(driver): 调整 CAN 收发器退出待机时序`

如发现当前改动同时涉及多个主题，应提示：

> 当前改动同时涉及驱动修复、日志调整和构建配置，建议拆分为多个 commit 后再执行提交。
