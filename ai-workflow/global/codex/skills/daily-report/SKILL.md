# Daily Report Skill

## 适用场景
当用户需要：
- 写日报
- 写周报
- 把零散工作记录整理成正式汇报
- 转成适合飞书发送的内容
- 将日报保存为固定文件

## 输入信息
尽量从以下信息中整理：
- 今日完成事项
- 调试/开发内容
- 提交代码内容
- 发现的问题
- 解决/推进情况
- 风险与阻塞
- 明日计划

## 输出要求
默认输出三部分：
1. 今日完成
2. 问题/风险
3. 明日计划

如果信息充分，再补充：
- 当前进度
- 需协同事项

## 输出风格
### 简洁版
适合飞书日报，控制在 5~10 行。

### 标准版
适合组内同步，结构清晰。

### 汇报版
适合面向领导，强调进展、结果、风险。

## 生成规则
1. 不虚构结果。
2. 未完成项要明确写“排查中/推进中”。
3. 语气专业，不口语化过度。
4. 技术细节太深时，日报中仅保留结论和影响。
5. 如果用户未指定格式，默认生成“标准版”。

## 文件命名规则
默认按日期命名，格式如下：

- `YYYY-MM-DD-daily-report.md`

例如：
- `2026-04-02-daily-report.md`
- `2026-04-03-daily-report.md`

如果同一天需要生成多个版本，可附加后缀：
- `YYYY-MM-DD-daily-report-v2.md`
- `YYYY-MM-DD-daily-report-leader.md`
- `YYYY-MM-DD-daily-report-feishu.md`

## 固定输出位置
默认输出到固定目录：

- `reports/daily/`

即最终文件路径格式为：

- `reports/daily/YYYY-MM-DD-daily-report.md`

例如：
- `reports/daily/2026-04-02-daily-report.md`

## 输出目录规则
1. 如果固定目录不存在，应先创建目录：
   - `reports/`
   - `reports/daily/`
2. 输出内容时，优先保存为 markdown 文件。
3. 如果用户明确指定其他目录，则以用户指定目录为准。
4. 如果当前项目已有日报目录规范，则优先遵循项目规范。

## 特别要求
1. 如果用户给的是零散聊天、日志、diff、便签，先归类再输出，不要原样堆叠。
2. 如果用户要求“生成并保存”，则应同时：
   - 生成日报正文
   - 按命名规则生成文件名
   - 输出到固定目录
3. 如果用户只要求“写日报”，则至少按上述命名和路径规则告知建议保存位置。
