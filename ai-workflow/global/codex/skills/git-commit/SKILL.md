# git-commit

## 目标

该技能用于管理如下 Git 工作流：

- 一个主分支：当前真正可 push 的业务分支
- 一个工作分支：AI 专用工作分支，命名跟随主分支变化
- AI 默认只在工作分支修改代码
- AI 修改完成后，不能自动 commit，必须先提示用户
- 当用户说“提交代码”时，不是直接 push，而是从工作分支列出候选 commit，供用户选择后合入主分支

本技能的目标是：

1. 尽量自动化处理分支创建、切换、候选 commit 管理
2. 保持主分支历史干净、可控、可挑选
3. 让 AI 改代码与正式提交解耦
4. 中文优先输出状态说明、风险提示和提交建议

---

## 工作流总览

### 分支模型

固定使用双分支模型：

- **主分支**：当前真正可 push 的业务分支
- **工作分支**：AI 专用工作分支

工作分支命名规则：

`ai/<清洗后的主分支名>-workspace`

示例：

- `develop` → `ai/develop-workspace`
- `feature/can_fix` → `ai/feature-can_fix-workspace`
- `release/v1.2` → `ai/release-v1.2-workspace`

### 主分支识别规则

- 当前分支如果 **不是** `ai/...-workspace`，则将其视为当前主分支
- 当前分支如果是 `ai/...-workspace`，则从分支名反推出对应主分支

### 主分支名清洗规则

当从主分支名生成工作分支名时：

- 将 `/` 替换为 `-`
- 将空格替换为 `-`
- 尽量保留原有可读性
- 前缀固定为 `ai/`
- 后缀固定为 `-workspace`

---

## 基本原则

### 1. AI 默认只在工作分支修改代码

只要 AI 要开始修改代码，就应优先工作在对应的工作分支：

- 如果当前在主分支：自动创建或切换到对应工作分支
- 如果当前已在对应工作分支：直接继续
- 如果当前在别的普通分支：把该分支视为主分支，再生成对应工作分支

### 2. 修改完成后，不能自动 commit

AI 完成修改后：

- 不得直接执行 commit
- 必须先向用户展示提交前摘要
- 由用户判断当前 bug / 需求是否已完成
- 只有用户确认后，才允许在工作分支执行 commit

### 3. “提交代码”默认不是 push

当用户说：

- 帮我提交代码
- 我要提交代码
- 准备交代码
- 选择要合入的提交

默认含义是：

> 从工作分支列出相对主分支新增的候选 commit，供用户选择后合入主分支

而不是：

- 直接在当前分支 commit
- 直接 merge 整个工作分支
- 直接 push 到远端

### 4. 工作分支保留历史

即使某些 commit 已经合入主分支：

- 不自动删除工作分支历史
- 不自动 reset 工作分支
- 不自动清空候选提交区

工作分支默认作为 AI 的长期候选提交区保留。

### 5. 不自动 push

无论是工作分支上的本地 commit，还是把候选 commit 合入主分支后：

- 默认不自动 push
- push 必须由用户明确决定

## 执行规则

## 场景一：开始处理 bug / 需求

当用户要求 AI 修改代码时，应按以下顺序处理：

1. 检查当前所在分支
2. 识别当前主分支
3. 计算对应工作分支名
4. 如果工作分支不存在，则自动创建
5. 切换到对应工作分支
6. 在工作分支上进行代码修改

### 特别规则：当前工作区不干净时

如果当前存在未提交改动：

- 不要静默切换分支
- 必须先提示用户当前工作区不干净
- 说明切换分支可能会把现有改动一起带到工作分支
- 待用户确认后再继续

提示应尽量简洁，例如：

- 当前分支
- 当前是否有未提交改动
- 切换后可能带来的影响
- 建议下一步动作

### 当工作分支不存在时

如果当前主分支第一次进入这套工作流，且对应工作分支不存在，默认允许自动执行：

- 创建工作分支
- 切换到工作分支

并告知用户：

- 原主分支是什么
- 新创建的工作分支是什么
- 后续 AI 修改将发生在工作分支，而不是主分支

---

## 场景二：AI 修改完成，准备在工作分支 commit

AI 修改完成后，不得自动 commit。

在执行 commit 前，必须先给出提交前摘要，至少包括：

- 当前主分支
- 当前工作分支
- 改动文件数量
- 主要改动摘要
- 文件较少时，可附上文件名
- 建议 commit message
- 是否发现明显无关文件
- 是否发现风险项（如编译产物、自动生成文件、临时调试代码等）

### 提示模板

建议使用如下风格：

准备在工作分支提交本次改动

- 主分支：<主分支名>
- 工作分支：<工作分支名>
- 改动文件：<数量>
- 主要改动：<简短摘要>
- 建议提交标题：<建议标题>
- 检查结果：<是否发现无关文件或风险>

请确认当前 bug / 需求是否已完成；确认后再执行提交。

### 只有在以下条件满足时才允许 commit

- 用户已明确确认当前任务已完成
- 当前改动主题相对单一
- 没有明显无关文件混入，或已向用户说明
- commit message 已生成且具备可读性

---

## 场景三：用户说“提交代码”

当用户说“提交代码”时，默认执行如下流程：

1. 找到当前主分支
2. 找到对应工作分支
3. 计算“工作分支相对主分支新增的 commit”
4. 以候选列表形式展示这些 commit
5. 支持用户先查看某个 commit 的详细 diff
6. 用户选择要合入的 commit
7. 切回主分支
8. 按原顺序逐个合入选中的 commit
9. 给出合入结果提示
10. 不自动 push

### 候选 commit 列表展示格式

默认展示：

- 序号
- 短 hash
- commit 标题

示例：

主分支：develop  
工作分支：ai/develop-workspace  

以下 commit 可从工作分支合入主分支：

1. a1b2c3d  fix(can): 修正收发器待机引脚控制逻辑
2. d4e5f6g  fix(bsp): 修复启动阶段空指针访问
3. h7i8j9k  refactor(uart): 拆分初始化与恢复流程

### 默认支持的用户操作

候选列表展示后，应允许用户继续表达以下意图：

- 合入 1 和 2
- 合入全部
- 只合入第 3 个
- 先看第 2 个详细 diff
- 先不要合入，继续开发

### 查看单个候选 commit 详细信息

如果用户要求查看某个候选 commit 的详细 diff，应提供：

- commit 标题
- 涉及的主要文件
- 改动摘要
- 必要时展示 diff 摘要

默认先展示摘要，不要无差别输出过长 diff，避免刷屏。

### 合入主分支的默认策略

用户选定候选 commit 后，应按如下方式处理：

- 切回主分支
- 按原顺序逐个合入选中的 commit
- 默认不 squash
- 默认不 reset 工作分支
- 默认不删除工作分支历史
- 默认不 push

合入后应给出简洁结果提示，例如：

- 已合入哪些 commit
- 当前所在主分支
- 是否有冲突
- 建议下一步（如测试、检查日志、手动 push）

---

## 风险控制

在以下情况下，不要直接执行高风险 Git 操作，应先提示用户：

- 当前工作区不干净，却准备切换分支
- 一次改动混入多个独立主题
- 提交中包含明显无关文件
- 检测到编译产物、二进制文件、自动生成文件
- 检测到临时调试日志、临时代码、临时宏
- 用户未明确要求，却要 push、rebase、reset、删除历史

### 嵌入式项目附加检查

如果仓库是嵌入式项目，提交前额外检查：

- 是否混入编译产物（如 `.bin`、`.elf`、`.map`、`.hex` 等）
- 是否混入自动生成代码或中间文件
- 是否误改芯片配置、时钟配置、引脚配置
- 是否保留临时日志、`printf`、调试开关
- 是否同时修改驱动、BSP、应用多个层次且缺乏明确理由

若检测到这些问题，应先提示，再由用户决定如何处理。

---

## 提交信息要求

提交信息生成时，应优先参考：

`shared/templates/commit-message.md`

要求：

- 中文优先
- 标题清晰可读
- 一眼能看懂改了什么
- 便于后续在候选列表中进行挑选
- 一个 commit 只表达一个主题

---

## 输出风格要求

所有说明、提示、建议尽量使用中文，风格应：

- 简洁
- 明确
- 少废话
- 可直接执行

建议输出结构：

1. 当前状态
2. 已执行动作
3. 当前建议
4. 风险提示（如有）
5. 建议提交标题或候选 commit 列表

避免：

- 只给命令，不解释
- 在风险操作前完全静默
- 在用户未确认前直接 commit
- 在用户未明确要求前直接 push
